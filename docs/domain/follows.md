# Follows domain

## Overview
The follows app lets collaborator accounts track athletes of interest. It
exposes endpoints for creating follow relationships, removing them, and listing
all tracked athletes for the authenticated collaborator, while enforcing plan
quotas on the number of follows.

## Data model
- **`Follow`** links a `Collaborator` to an `Athlete`, storing notification
  preferences for news, stats, and contracts. Unique and index constraints
  prevent duplicates and optimise lookup by athlete or collaborator.

## Serializers
`FollowSerializer` serializes follow records with the nested
`AthletePublicSerializer`, returning read-only fields for downstream consumers
(messaging, dashboards, etc.).

## Permissions and entitlements
- `IsCollaboratorUser` ensures only authenticated users with at least one
  collaborator membership may interact with the API.
- `AthleteFollowView` enforces entitlement limits with
  `get_collaborator_plan_features()` and `user_feature_requirement("follow_slots")`.
  When `max_follows` is exceeded or the feature is unavailable, the view returns
  a structured denial generated by `requirement_denied_payload()`.
- All endpoints require authentication; staff bypass collaborator membership
  checks implicitly.

## API surface
Routes are mounted under `/api/`:

| Method | Route | Description | Auth |
| --- | --- | --- | --- |
| `POST` | `/athletes/<athlete_id>/follow/` | Follow an athlete for the current collaborator, respecting follow quotas. | Collaborator |
| `DELETE` | `/athletes/<athlete_id>/follow/` | Unfollow the athlete. | Collaborator |
| `GET` | `/me/follows/` | List follow records across all of the user's collaborator memberships. | Collaborator |

Both create and delete operations accept an optional `collaborator_id` to select
which membership is used; otherwise the first membership for the user is chosen.
